# BLE Data Whitening on the nRF24L01+ — Bugs, Correct Implementation, and Lessons

## 1. Overview

BLE applies **data whitening** to every on-air packet (PDU + CRC) before
transmission.  Whitening XORs the payload with a pseudo-random bit sequence
generated by a 7-bit Linear Feedback Shift Register (LFSR).  The receiver
applies the identical sequence to recover the original data.

### Why BLE uses whitening

- **Reduce DC bias** — Without whitening, a long run of identical bits would
  produce a constant RF carrier, concentrating energy at the centre frequency
  and degrading receiver performance.
- **Improve clock recovery** — The whitened bit stream has frequent transitions,
  making it easier for the receiver's clock-recovery PLL to stay synchronised.
- **Spectral spreading** — Energy distributes more evenly across the channel
  bandwidth instead of clumping at discrete frequencies.

### The polynomial

BLE uses the polynomial **x^7 + x^4 + 1**, which defines the LFSR feedback
taps.  In the 7-bit Galois LFSR state, this means: when the shift-out bit is
1, XOR the state with bits at positions 4 and 0 before the next shift.  The
encoded tap mask in the 8-bit working register is `0x11` (bit 4 = x^4, bit 0
= 1).

### The 7-bit LFSR mechanism

The LFSR is initialised with a **channel-dependent seed** before each packet.
It then shifts once per bit, producing one whitening bit per shift.  Over a
39-byte maximum advertising PDU + CRC, the LFSR shifts 312 times.

## 2. How BLE Whitening Works (Correct Implementation)

### 2.1 LFSR state diagram — Galois form, left-shifting

```
          ┌──┬──┬──┬──┬──┬──┬──┬──┐
 bit pos: │ 7│ 6│ 5│ 4│ 3│ 2│ 1│ 0│
          └──┴──┴──┴──┴──┴──┴──┴──┘
                ↑              ↑  ↑
                │              │  │
                └──feedback────┘  │
                    (XOR 0x11)    │
                                   └── also part of feedback
         shift direction: ← ← ← ← ← ← ← ←
```

At each clock:

1. **Check bit 7** (the MSB of the 8-bit working register).  
   - If bit 7 = 1: XOR the register with `0x11` (taps at bits 4 and 0),
     then output a whitening bit of 1 for this position.
   - If bit 7 = 0: no XOR, output a whitening bit of 0.
2. **Left-shift** the register by 1 (`lfsr <<= 1`), with the 8-bit `uint8_t`
   naturally discarding the bit that shifted out of bit 7.

The output whitening bit is used to XOR the corresponding data bit.

### 2.2 Seed derivation from channel index

The Bluetooth Core Spec (Vol 6 Part B §3.2) states the 7-bit LFSR seed is
"the value of the channel index plus 64."  In the spec's LSB-first bit
ordering, this means position 1 is always 1, and positions 6:1 carry the
6-bit channel index value.

For the nRF24L01+, which delivers bytes MSbit-first, we must **bit-swap the
channel index** before loading it into the left-shifting Galois LFSR.  This
converts the spec's LSB-first seed layout into the MSB-first layout the LFSR
expects.

The resulting seed formula, per Dmitry Grinberg's reference implementation:

```
seed = swapbits(channel_idx) | 2
```

- `swapbits(channel_idx)` — bit-reverses the channel index, placing its bits
  in the MSB-first positions of the LFSR.
- `| 2` — sets bit 1, which is position 1 in the Galois register (the
  constant-1 part of the spec's "channel index + 64" formula).

**Derivation step by step for channel 37 (0x25):**

```
channel = 37 = 0x25 = 0b00100101

Spec's seed (LSB-first, 7-bit):
  position:  1  2  3  4  5  6  7
  value:     1  1  0  1  0  0  1    (= 64 + 37 = 101 in the spec's view)

Our LFSR shifts left (MSB-first), so we load the bits in reverse:
  swapbits(0x25) = 0xA4 = 0b10100100

Set bit 1 (position 1 in the Galois register):
  0xA4 | 0x02 = 0xA6 = 0b10100110

Seed = 0xA6
```

### 2.4 Full TX encoding pipeline (Dmitry Grinberg's btLePacketEncode)

When **transmitting** a BLE packet from the nRF24L01+, the complete pipeline
from plaintext PDU to the bytes written into the TX FIFO is:

1. **Compute CRC24** over the PDU (header + payload) using polynomial
   `0x00065B` and initial value `0x555555`.
2. **Bit-swap the 3 CRC bytes** — the CRC is computed in LSB-first order but
   must be transmitted MSB-first via the nRF24.
3. **Whiten** PDU + CRC with the Galois LFSR using the channel's seed.
4. **Bit-swap every byte** of the entire whitened packet — the LFSR produces
   whitening bits in LSB-first order, but the nRF24 SPI interface expects
   MSB-first bytes.

```
Plaintext PDU ──→ CRC24 ──→ swapbits(CRC) ──→ Whiten(LFSR) ──→ swapbits(all) ──→ nRF24 TX FIFO
```

### 2.5 Full RX decoding pipeline (nRF24 sniffer)

When **receiving** a BLE packet from the nRF24L01+, the pipeline reverses:

1. **Bit-swap each received byte** — the nRF24 RX FIFO contains MSB-first
   bytes; BLE protocol processing requires LSB-first.
2. **Dewhiten** with the same Galois LFSR and channel seed.
3. **Bit-swap the 3 CRC bytes** — CRC is transmitted MSB-first but verified
   LSB-first.
4. **Verify CRC24** over the PDU.

```
nRF24 RX FIFO ──→ swapbits(all) ──→ Dewhiten(LFSR) ──→ swapbits(CRC) ──→ Verify CRC24 ──→ Plaintext PDU
```

The library function `nrf24::ble::dewhiten()` encapsulates steps 1 and 2
(bit-swap + LFSR dewhiten) in a single call, so the caller never needs to
remember the bit-swap step.

### 2.6 Code walkthrough — the correct Galois LFSR

```
@code
 // Galois LFSR dewhitening — symmetric (same function whitens and dewhitens)
 // Polynomial: x^7 + x^4 + 1, encoded as 0x11 in the 8-bit working register
 // Reference: Dmitry Grinberg, "Bit-banging Bluetooth Low Energy"
 
 void dewhiten(uint8_t *data, uint8_t len, uint8_t channel_idx)
 {
     // Step 1: bit-swap each byte (nRF24 MSbit-first → BLE LSbit-first)
     for (uint8_t i = 0; i < len; i++)
         data[i] = nrf24::ble::swapbits(data[i]);
 
     // Step 2: Galois LFSR
     uint8_t lfsr = nrf24::ble::swapbits(channel_idx) | 0x02;  // seed
     for (uint8_t i = 0; i < len; i++) {
         for (uint8_t m = 1; m; m <<= 1) {  // m walks bit 0→7; stops when wraps
             if (lfsr & 0x80) {              // bit 7 set?
                 lfsr ^= 0x11;              // apply feedback taps (bits 4, 0)
                 data[i] ^= m;             // XOR whitening bit into data
             }
             lfsr <<= 1;                    // left-shift (Galois)
         }
     }
 }
 @endcode
```

Key points:
- `m` starts at 1 (bit 0) and walks through 1, 2, 4, 8, 16, 32, 64, 128,
  then wraps to 0 (since `uint8_t` 128 << 1 = 0), terminating the loop.
- The XOR of the whitening bit into `data[i]` happens **inside** the bit-7
  check, before the shift.  This is how Dmitry Grinberg's original code
  works: the whitening bit is 1 when the LFSR output is 1, and 0 otherwise.
- XOR whitening is symmetric: the same function whitens and dewhitens.

## 3. What Went Wrong — Bug Analysis

Two independent bugs existed in the original `dewhiten()` implementation.
Both must be fixed together; neither alone produces correct results.

### 3.1 Bug #1: Missing bit-swap of data bytes before dewhitening

**Root cause:** The project's earlier learning document contained the
incorrect claim that the bit order within a byte "matches" between nRF24 and
BLE:

> "Bits within byte: LSB first = LSB first = Both match!"

This is **wrong**.  The nRF24L01+ Product Specification §7.3 states that the
packet format has "MSB to the left," meaning the nRF24 transmits and receives
MSbit-first within each byte.  The BLE specification (Vol 6 Part B §1.3.1)
specifies LSbit-first transmission.  Additionally, the nRF24's preamble
auto-selection logic confirms MSB-first operation: it chooses between 0xAA
and 0x55 based on the MSbit of the first data byte.

**Impact:** Without the bit-swap step, every byte from the nRF24 RX FIFO is
processed in the wrong bit order.  The LFSR XOR is applied to the wrong bits,
producing completely incorrect dewhitened data.  Some bytes may appear correct
by coincidence when the byte value is **bit-palindromic** (equal to its own
bit-reversal, such as `0x42`, `0x18`, `0x81`, `0xFF`, `0x00`), but this is
purely accidental.

**Concrete example:** A correct BLE PDU header byte `0x40` on channel 37:
- After correct whitening: `0xCD`
- After bit-swap for nRF24 TX FIFO: `0xB3`
- nRF24 RX FIFO contains: `0xB3`
- **Correct** dewhiten (swap first): `swapbits(0xB3) = 0xCD`, then LFSR → `0x40`
- **Buggy** dewhiten (no swap): LFSR applied to `0xB3` → `0x3E` (wrong)

### 3.2 Bug #2: Wrong LFSR algorithm (Fibonacci instead of Galois)

**Root cause:** The implementation used a right-shifting **Fibonacci** LFSR
instead of the correct left-shifting **Galois** LFSR.  These two LFSR forms
produce **different output sequences** for the same polynomial, because the
feedback is applied at different points in the shift cycle.

Additionally, the seed was wrong: `(channel_idx & 0x3F) | 0x40` instead of
`swapbits(channel_idx) | 2`.  These are not equivalent:

| Channel | Wrong: `(ch & 0x3F) \| 0x40` | Correct: `swapbits(ch) \| 2` |
|---------|-----------------------------|-----------------------------|
| 37 | `0x25 & 0x3F = 0x25`, `0x25 \| 0x40 = 0x65` | `swapbits(0x25) = 0xA4`, `0xA4 \| 0x02 = 0xA6` |
| 38 | `0x26 & 0x3F = 0x26`, `0x26 \| 0x40 = 0x66` | `swapbits(0x26) = 0x64`, `0x64 \| 0x02 = 0x66` |
| 39 | `0x27 & 0x3F = 0x27`, `0x27 \| 0x40 = 0x67` | `swapbits(0x27) = 0xE4`, `0xE4 \| 0x02 = 0xE6` |

Note that channel 38 coincidentally produces the same seed (`0x66`) by both
formulas — but the Fibonacci and Galois LFSRs with different feedback XOR
values still produce completely different whitening sequences.

**All five differences between the buggy and correct implementations:**

| Aspect | Buggy (Fibonacci) | Correct (Galois) |
|--------|-------------------|------------------|
| Shift direction | Right (`>>= 1`) | Left (`<<= 1`) |
| Feedback check | Bit 0 (`& 0x01`) | Bit 7 (`& 0x80`) |
| Feedback XOR | `0x48` | `0x11` |
| Seed | `(ch & 0x3F) \| 0x40` | `swapbits(ch) \| 2` |
| Bit-swap | **Missing** | Performed first |

**Concrete example — different whitening sequences for channel 37:**

For the first byte of data, the two LFSRs produce these XOR masks:

| LFSR form | Seed | First-byte XOR mask | Whitened `0x40` |
|-----------|------|---------------------|------------------|
| Galois (correct) | `0xA6` | `0x8D` | `0x40 ^ 0x8D = 0xCD` |
| Fibonacci (buggy) | `0x65` | `0x4D` | `0x40 ^ 0x4D = 0x0D` |

The whitened outputs are completely different (`0xCD` vs `0x0D`), and no
amount of fiddling with the seed can make the Fibonacci form produce the
same sequence as the Galois form — they are fundamentally different
transformations.

## 4. How It Was Corrected — Code Breakdown

### 4.1 The corrected `dewhiten()` function

```
@code
 void dewhiten(uint8_t *data, uint8_t len, uint8_t channel_idx)
 {
     // Step 1: bit-swap each byte (nRF24L01+ MSbit-first → BLE LSbit-first)
     for (uint8_t i = 0; i < len; i++)
         data[i] = nrf24::ble::swapbits(data[i]);
 
     // Step 2: Galois LFSR dewhitening (Dmitry Grinberg's algorithm).
     // Polynomial x^7+x^4+1 encoded as 0x11 in the 7-bit state.
     // Seed: nrf24::ble::swapbits(channel_idx) | 2.
     uint8_t lfsr = nrf24::ble::swapbits(channel_idx) | 0x02;  // line A
     for (uint8_t i = 0; i < len; i++) {                        // line B
         for (uint8_t m = 1; m; m <<= 1) {                      // line C
             if (lfsr & 0x80) {                                  // line D
                 lfsr ^= 0x11;                                    // line E
                 data[i] ^= m;                                   // line F
             }
             lfsr <<= 1;                                          // line G
         }
     }
 }
 @endcode
```

**Line-by-line annotation:**

| Line | Code | Explanation |
|------|------|-------------|
| A | `lfsr = swapbits(channel_idx) \| 0x02` | Compute channel-specific Galois seed. `swapbits` reverses bit order to match MSB-first LFSR. `\| 0x02` sets position 1. |
| B | `for (uint8_t i = 0; i < len; i++)` | Process each byte of the (already bit-swapped) data. |
| C | `for (uint8_t m = 1; m; m <<= 1)` | Walk through bit positions 0–7. `m` starts at 1, doubles each iteration, wraps to 0 after bit 7 (terminates). |
| D | `if (lfsr & 0x80)` | Check if the LFSR output bit (MSB of the 8-bit working register) is 1. |
| E | `lfsr ^= 0x11` | Apply Galois feedback: XOR the polynomial taps (bits 4 and 0) into the state. |
| F | `data[i] ^= m` | XOR the whitening bit (1) into the current bit position of the data byte. |
| G | `lfsr <<= 1` | Left-shift the LFSR state (Galois form). `uint8_t` overflow naturally discards the bit shifted out of position 7. |

### 4.2 The `swapbits()` utility function

```
@code
 inline uint8_t nrf24::ble::swapbits(uint8_t v)
 {
     uint8_t r = 0;
     if (v & 0x80) r |= 0x01;  // bit 7 → bit 0
     if (v & 0x40) r |= 0x02;  // bit 6 → bit 1
     if (v & 0x20) r |= 0x04;  // bit 5 → bit 2
     if (v & 0x10) r |= 0x08;  // bit 4 → bit 3
     if (v & 0x08) r |= 0x10;  // bit 3 → bit 4
     if (v & 0x04) r |= 0x20;  // bit 2 → bit 5
     if (v & 0x02) r |= 0x40;  // bit 1 → bit 6
     if (v & 0x01) r |= 0x80;  // bit 0 → bit 7
     return r;
 }
 @endcode
```

Uses the if-else cascade form from Dmitry Grinberg's original code because it
is the most portable across embedded toolchains — no table lookups, no
architecture-specific intrinsics.

## 5. Reference Implementation Comparison

### 5.1 Dmitry Grinberg's btLeWhiten

| Aspect | Dmitry Grinberg (original) | Our `nrf24::ble::dewhiten()` |
|--------|-----------------------------|------------------------------|
| Language | C (AVR GCC) | C++ (ESP-IDF) |
| Seed | `swapbits(chan) \| 2` | `nrf24::ble::swapbits(channel_idx) \| 0x02` |
| LFSR form | Galois, left-shift | Galois, left-shift |
| Feedback XOR | `0x11` | `0x11` |
| Bit order | LSB→MSB (`m = 1; m; m <<= 1`) | Same |
| Bit-swap | Separate: `swapbits` before/after in `btLePacketEncode` | Integrated: first step inside `dewhiten()` |
| Symmetry | Same function whitens and dewhitens | Same |

**Result: identical output for identical inputs.** Our implementation
encapsulates the bit-swap inside `dewhiten()` rather than requiring the
caller to do it separately, but the algorithm is identical.

### 5.2 omriiluz/NRF24-BTLE-Decoder BTLEWhiten

The omriiluz implementation iterates the XOR mask from MSB to LSB
(`m = 0x80; m; m >>= 1`) instead of LSB to MSB.  However, it also generates
the whitening sequence in the same direction (bit 7 check first), and the
XOR mask bit that is set corresponds to the **same data bit position** in both
implementations:

- When the LFSR output is 1 at step *k*, both set bit *k* of the mask byte.
- The only difference is the iteration order within the inner loop, which
  does not affect the final XOR mask.

**Result: identical output** despite the reversed byte-mask iteration
direction.  The crucial invariant is that each LFSR clock step is associated
with one data bit, and both implementations maintain that correspondence.

## 7. Pitfalls and Lessons Learned

### 7.1 "Both match!" claims about bit order must be verified against datasheets

The original learning document stated that nRF24 and BLE bit order "match."
This was never verified against the nRF24L01+ datasheet.  The datasheet is
unambiguous: "packet format with MSB to the left" (§7.3), and preamble
auto-selection based on the MSbit of the first data byte confirms MSB-first
operation.

**Always read the datasheet.  Never assume two protocols have compatible
bit ordering based on intuition.**

### 7.2 Galois vs Fibonacci LFSR produce different sequences

The polynomial alone does not determine the LFSR output.  The same polynomial
x^7 + x^4 + 1 can be implemented as either a Galois or Fibonacci LFSR, and
these produce **different** whitening bit sequences.  The BLE specification
defines a Galois-form LFSR.  Using the Fibonacci form produces entirely
wrong dewhitening, even if the polynomial, seed, and initial state appear
"equivalent."

**When implementing an LFSR from a specification, verify the LFSR form (Galois
vs Fibonacci, shift direction, feedback position) against the specification
or reference implementation.**

### 7.3 The LFSR seed requires bit-swap of the channel index

The Bluetooth Core Spec states the seed is "channel index + 64."  This is
described in the spec's LSBit-first bit ordering.  For the nRF24 (MSB-first),
the correct seed is `swapbits(channel) | 2`, which is **not** the same as
`(channel & 0x3F) | 0x40` (the literal interpretation of the spec formula).

**Bit-order conventions affect seed computation.  Use the reference
implementation's seed formula, not the spec's literal formula, when working
across bit-order boundaries.**

### 7.4 Unit tests for whitening/dewhitening must include round-trip and known-vector checks

The original bug was not caught by testing because there were no round-trip
tests.  A simple `whiten(dewhiten(data)) == data` test would have immediately
revealed both bugs.

**Mandatory test coverage for whitening:**
- Round-trip for every supported channel
- Known whitening vectors from a trusted reference implementation
- Cross-channel independence (wrong channel → different output)
- Edge cases: channel 0 (minimum seed), channel 39 (maximum)

### 7.5 When borrowing algorithms, preserve the complete implementation

The seed formula, LFSR form, feedback polynomial, and bit-swap steps form an
inseparable unit.  Mixing parts from different implementations (e.g., using
the spec's literal seed formula with Grinberg's LFSR code, or copying the
LFSR but omitting the bit-swap) will produce incorrect results.

**Borrow complete, tested implementations.  Do not cherry-pick individual
components from different sources and assume they are interchangeable.**
